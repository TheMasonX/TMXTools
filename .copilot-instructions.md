# TMXTools Copilot Instructions

This document provides guidance for AI assistants (including GitHub Copilot) working on the TMXTools project.

## Project Overview

**TMXTools** is a modern .NET 8 library providing utilities and extensions for C# and WPF development.

- **TMXTools**: Core cross-platform library (.NET 8)
- **TMXTools.WPF**: WPF extensions and controls (.NET 8-Windows)
- **Test Projects**: NUnit-based unit tests with code coverage

## Architecture & Design Principles

### Core Principles

1. **Composition over Inheritance**: Prefer interfaces and composition
2. **Explicit Error Handling**: Always handle errors clearly, no silent failures
3. **Guard Clauses**: Use early returns for validation
4. **Null Safety**: Use `is null` and `is not null` patterns
5. **Async-First**: All I/O operations should be async
6. **Dependency Injection**: Enable testability and flexibility

### Code Organization

```
Feature/
├── Interfaces/
│   └── I[Feature].cs
├── Models/
│   └── [Feature]Model.cs
├── [FeatureUtils].cs
└── Extensions/
    └── [Feature]Extensions.cs

Tests/
├── [Feature]Tests.cs
└── Fixtures/
    └── [Feature]Fixture.cs
```

## Coding Standards

### Style & Formatting

- **EditorConfig**: All formatting rules defined in `.editorconfig`
- **Naming**: PascalCase for types/members, camelCase for locals, _camelCase for private fields
- **Async Methods**: Always suffix with `Async`
- **Interfaces**: Always prefix with `I`

See `STYLE_GUIDE.md` for detailed conventions.

### Required Patterns

**Async/Await:**
```csharp
public async Task<T> DoWorkAsync(CancellationToken cancellationToken = default)
{
    var result = await GetDataAsync().ConfigureAwait(false);
    cancellationToken.ThrowIfCancellationRequested();
    return result;
}
```

**Error Handling:**
```csharp
public void ValidateInput(string? input)
{
    if (string.IsNullOrWhiteSpace(input))
        throw new ArgumentException("Input cannot be empty", nameof(input));
}
```

**Null Checking:**
```csharp
if (value is null)
    return;

if (value is not null)
    Process(value);
```

**Guard Clauses:**
```csharp
public void Process(string input)
{
    if (string.IsNullOrEmpty(input))
        return;
    
    // Main logic starts here
}
```

## Testing Requirements

### When to Write Tests

✅ **Always test:**
- Public APIs
- Error conditions
- Async methods
- Complex logic
- Edge cases

### Test Structure

All tests follow the **AAA Pattern**:
```csharp
[Test]
public async Task Method_Scenario_Expectation()
{
    // Arrange: Setup test data
    var input = PrepareTestData();

    // Act: Execute the code
    var result = await sut.MethodAsync(input);

    // Assert: Verify results
    Assert.That(result, Is.EqualTo(expected));
}
```

### Test Naming

Format: `[MethodName]_[Scenario]_[ExpectedOutcome]`

✅ Good:
- `ReadAsync_WithValidPath_ReturnsContent`
- `Parse_WithInvalidInput_ThrowsArgumentException`
- `Calculate_WithZeroDivisor_ThrowsDivideByZeroException`

❌ Avoid:
- `TestReadAsync`
- `Test_Read`
- `ReadAsyncWorks`

### Coverage Expectations

- **TMXTools**: Aim for 80%+ coverage
- **TMXTools.WPF**: Aim for 75%+ coverage (excluding pure UI)
- **Focus on behavior**, not line count

## Documentation Requirements

### XML Documentation (Required for Public APIs)

```csharp
/// <summary>
/// Brief description of what this does.
/// </summary>
/// <param name="paramName">Description of the parameter.</param>
/// <param name="cancellationToken">Token to cancel the operation.</param>
/// <returns>Description of the return value.</returns>
/// <exception cref="ArgumentException">When arguments are invalid.</exception>
/// <remarks>
/// Additional context about behavior, performance, etc.
/// </remarks>
/// <example>
/// <code>
/// var result = await method(input);
/// </code>
/// </example>
public async Task<T> MethodAsync(string param, CancellationToken cancellationToken = default)
{
    // Implementation
}
```

### When NOT to Comment

❌ Avoid comments that restate the code:
```csharp
// Bad: Comment is redundant
counter++;  // Increment counter
```

✅ Good: Comments explain **why**:
```csharp
// Use exponential backoff to avoid overwhelming the server
await Task.Delay(TimeSpan.FromSeconds(Math.Pow(2, retryCount)));
```

## Common Patterns

### Interface-Based Design

```csharp
// Good: Depends on abstraction
public class Service
{
    private readonly IFileReader _reader;
    public Service(IFileReader reader) => _reader = reader;
}

// Avoid: Depends on concrete implementation
public class Service
{
    public void Process()
    {
        var content = FileUtils.ReadAllText(path);
    }
}
```

### CancellationToken Support

Always include `CancellationToken` parameter with default:
```csharp
public async Task<string> LoadAsync(string path, CancellationToken cancellationToken = default)
{
    var data = await File.ReadAllTextAsync(path, cancellationToken).ConfigureAwait(false);
    cancellationToken.ThrowIfCancellationRequested();
    return data;
}
```

### Property Initialization

```csharp
// Prefer auto-properties
public string Name { get; set; } = "";

// Prefer init-only for immutability
public string Path { get; init; } = ".";

// Use records for data containers
public record FileInfo(string Path, long Size, DateTime Modified);
```

## What NOT to Do

❌ **Never:**
- Use `var` for unclear types
- Block async code with `.Result` or `.Wait()`
- Silently catch exceptions
- Create `async void` methods (except event handlers)
- Mix sync and async without clear intent
- Use public fields instead of properties
- Create deep inheritance hierarchies
- Ignore cancellation tokens
- Leave TODO comments without context

❌ **Avoid:**
- Nullable reference types without handling
- Creating new interfaces for single implementations
- Complex LINQ chains without explanation
- Hard-coded values that should be constants
- `static` everything
- Shared state without synchronization

## File Handling

### Proper File Operations

```csharp
// Good: Using statement, async, error handling
public async Task<string> ReadFileAsync(string path)
{
    if (!File.Exists(path))
        throw new FileNotFoundException($"File not found: {path}");

    try
    {
        return await File.ReadAllTextAsync(path).ConfigureAwait(false);
    }
    catch (IOException ex)
    {
        throw new InvalidOperationException($"Failed to read {path}", ex);
    }
}

// Avoid: Not async, no error handling
public string ReadFile(string path)
{
    return File.ReadAllText(path);
}
```

## When Creating New Code

### Checklist

- [ ] Does it follow the naming conventions in `STYLE_GUIDE.md`?
- [ ] Is it async (for I/O operations)?
- [ ] Does it support `CancellationToken`?
- [ ] Are public APIs documented with XML comments?
- [ ] Are there unit tests?
- [ ] Does it handle errors explicitly?
- [ ] Does it follow the composition pattern?
- [ ] Is it testable (dependencies injected)?
- [ ] Does it conform to `.editorconfig`?
- [ ] Are there no TODO comments left behind?

### Code Review Criteria

When reviewing code, verify:
1. ✅ Code style matches `.editorconfig`
2. ✅ Tests exist and pass
3. ✅ Public APIs are documented
4. ✅ No blocking async calls
5. ✅ Error handling is explicit
6. ✅ No silent failures
7. ✅ Performance is reasonable
